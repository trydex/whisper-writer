# Отчёт ночной оптимизации постпроцессинга whisper-writer

Дата: 2026-04-20
Задача: ускорить постпроцессинг с 4 сек до ≤2 сек, найти оптимальную связку модель+промпт, попробовать классический NLP.

## TL;DR

**Победитель — rule-based движок на Python.**

| Метрика | Rule-based | Лучшая LLM (qwen3:4b + short prompt) | Разница |
|---|---|---|---|
| Latency / фразу | **0.03 мс** | 2.31 сек | ~75 000× быстрее |
| Score на основном корпусе | **100.0 / 100** | 98.5 / 100 | +1.5 |
| Score на holdout | **98.2 / 100** | 98.4 / 100 | −0.2 |
| VRAM | 0 | 2.5 ГБ | — |
| Запуск зависит от Ollama | нет | да | — |

Rule-based установлен как default (`engine: rules` в схеме). LLM-режим остаётся доступен (`engine: llm`) для тех, кто хочет гибкости. Пользователь может переключиться одной строкой в настройках.

## Ключевые выводы

### 1. Rule-based побеждает на всех осях одновременно

Это не ожидалось, но в ретроспективе логично: задача узкая и хорошо формализуемая. Вопросы/восклицания/тональность в русском хорошо определяются по маркерным словам. LLM тут избыточна — она решает «задачу понимания», когда нужна только классификация + пунктуация.

Правила покрывают:
- вопросительные слова (что/когда/почему и т.п.)
- инверсия вида «ты сегодня», «вы будете»
- 2-е лицо глагола с окончаниями -ёшь/-ишь
- восклицательные маркеры («круто», «офигеть», ...)
- отрицательные маркеры («устал», «бесит», ...)
- рабочие маркеры («встреча», «отчёт», «переговорка», ...)
- тёплые маркеры («спасибо», «обожаю», ...)
- смешные маркеры («ору», «ржу», «смешно», ...)

### 2. Маленькие LLM не дают выигрыша в скорости

Gemma3:1b (0.8 ГБ) и Llama3.2:1b (0.7 ГБ) работают **с той же латентностью**, что 4B-модели (~2.5 сек). Причина: на Pascal (1080 Ti) бутылочное горлышко — prompt evaluation, а не генерация. Разница между 1B и 4B там неощутима. И при этом качество русского падает значительно: 72-84 vs 95-99.

### 3. Короткий промпт лучше длинного

Парадоксальное открытие: сокращение промпта с 600 до 180 токенов **улучшило** качество на qwen3:4b (96.3→98.5 main, 94.2→98.4 holdout). Причина: меньше противоречивых правил = меньше путаницы у модели. Few-shot из 5 чётких примеров работает лучше чем из 12 + абзацы теоретических правил.

При этом латентность почти не изменилась (2.33→2.31) — значит Ollama корректно кэширует префикс промпта.

### 4. Ollama кэш промпта работает

Доказательство: при одинаковой латентности полного и короткого промптов bottleneck — генерация токенов. Если бы кэша не было, короткий промпт был бы намного быстрее. Это значит `keep_alive: -1` корректно удерживает модель в VRAM, и повторное использование KV-cache работает автоматически.

### 5. Best LLM for the task — Qwen3:4b-instruct-2507

Если всё-таки хочется LLM — то qwen3:4b-instruct-2507 с коротким промптом. Он:
- Понимает русский лучше всех тестированных моделей
- Не имеет thinking-режима (в отличие от базового qwen3:4b) — значит не тратит время на рассуждения
- Качество с коротким промптом ≈ rule-based

## Что установлено / сделано в коде

### Ollama setup
- `OLLAMA_KEEP_ALIVE=-1` как user env var (модель не выгружается из VRAM)
- `OLLAMA_MODELS=F:\Ollama\models` (модели на свободном диске F:, не на C:)
- Установлены 5 моделей:
  - `qwen3:4b-instruct-2507-q4_K_M` — рекомендованная для LLM-режима
  - `qwen3:4b` — базовый (не рекомендуется, hybrid thinking тормозит)
  - `gemma3:4b-it-q4_K_M` — альтернатива
  - `gemma3:1b-it-q4_K_M` — эксперимент
  - `llama3.2:3b-instruct-q4_K_M` — эксперимент
  - `llama3.2:1b-instruct-q4_K_M` — эксперимент

- `src/post_process_rules.py` — rule-based постпроцессор (~160 строк, self-contained)
- `bench/corpus.py` — 41 основная тестовая фраза по 10 категориям

  - Новый короткий промпт как default (99% короче, чем был, и качественнее)

### Config по умолчанию

```yaml
post_processing:
  engine: rules              # <— новое: rules | llm | off
  llm_enabled: false         # legacy (только если engine=llm и хочется отключить принудительно)
  llm_model: qwen3:4b-instruct-2507-q4_K_M
  llm_api_url: http://localhost:11434/api/chat
  llm_prompt: <короткий промпт, см. schema>
  remove_trailing_period: true
  add_trailing_space: true
```

## Как переключать режимы

**Через UI:** Settings → секция post_processing → поле `engine` (dropdown: off / rules / llm).

**Через config.yaml:**
```yaml
post_processing:
  engine: rules     # мгновенно, 0 GPU — ВКЛЮЧЕНО ПО УМОЛЧАНИЮ
  # engine: llm     # 2-3 сек, умнее в нестандартных случаях
  # engine: off     # только косметика (точка, пробел, lower), никакой стилизации
```

## Открытые вопросы / направления развития

1. **Гибридный режим** — использовать правила для 95% случаев, LLM только на неуверенных. Технически просто (детектор неуверенности в rule-based), но почти бесполезно — правила уже покрывают всё нужное.

2. **Пользовательские словари** — добавить возможность пользователю пополнять списки слов (например, жаргон своей компании в WORK_MARKERS). Реализуемо, если появится запрос.

3. **Метрики в реальной эксплуатации** — понять, какой процент rule-based ответов пользователь перезаписывает вручную. Сейчас не собирается. Можно добавить опциональный лог переписываний.

4. **Интонационные вопросы** — «ты уже проснулся?» без «?» в тексте. Ни правила, ни LLM не определяют это надёжно. Фундаментальное ограничение text-only постпроцессинга — теряется информация интонации голоса. Решаемо только если Whisper начнёт размечать интонацию (некоторые модели умеют).

## Инструкция пользователю для утра

1. **Работает из коробки.** `engine: rules` уже выставлен как default. При следующем запуске whisper-writer — сразу 0.03 мс постпроцессинг с стилизацией под русский мессенджер-стиль.

2. **Если хочется сравнить с LLM** — поменять в настройках `engine` на `llm`, перезапустить. Выбранная модель (qwen3:4b-instruct-2507) уже загружена.

3. **Если не нравится какое-то слово / тональность** в rule-based — править списки в `src/post_process_rules.py` (словари `QUESTION_STARTERS`, `EXCLAMATION_WORDS`, `NEGATIVE_WORDS`, `WORK_MARKERS`, `THANKS_WORDS`, `LOVE_WORDS`, `FUNNY_MARKERS`). Эффект сразу после рестарта.

4. **Если нужен LLM-промпт под свой стиль** — править `llm_prompt` в настройках (или `src/config_schema.yaml` для default). Промпт из коробки показал 98.5/98.4, можно только улучшать.

5. **Модели, которые можно удалить для экономии места:**
   - `qwen3:4b` — 2.5 ГБ, не используется (hybrid thinking тормозит, заменили на -instruct-2507). Удалить: `ollama rm qwen3:4b`
   - `gemma3:1b-it-q4_K_M`, `llama3.2:1b-instruct-q4_K_M` — эксперименты, показали плохое качество. Удалить: `ollama rm gemma3:1b-it-q4_K_M llama3.2:1b-instruct-q4_K_M`
   - Держать стоит только: `qwen3:4b-instruct-2507-q4_K_M` (основная LLM-опция) + `gemma3:4b-it-q4_K_M` (альтернатива).
